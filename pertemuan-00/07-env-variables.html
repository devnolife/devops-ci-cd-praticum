Environment Variables

Satu kode, banyak pengaturan — rahasia konfigurasi yang bikin aplikasi kamu fleksibel di mana pun dijalankan.

🎯 Analogi Sederhana

🏢 AC di Gedung Kampus

Bayangkan AC di gedung kampus kamu. Mesin AC-nya (kode program) sama persis di semua lantai. Tapi suhu yang diatur di setiap lantai beda:

  • Lantai 1 (Development) → 24°C — nyaman buat ngoding
  • Lantai 2 (Staging) → 22°C — agak dingin buat testing
  • Lantai 3 (Production) → 20°C — stabil dan terkontrol

Kamu mengubah pengaturan suhu (environment variable), BUKAN mesin AC-nya (kode). Itulah inti dari env var!

🧁 Resep Kue

Atau bayangkan resep kue: resepnya sama persis, tapi bahan yang dipakai bisa berbeda — coklat lokal vs coklat impor, gula pasir vs gula aren. Hasilnya mirip, tapi disesuaikan dengan "lingkungan" (bahan yang tersedia). Environment variable = bahan-bahannya.

📖 Apa itu Environment Variable?

Environment variable (atau env var) adalah nilai konfigurasi yang disimpan di luar kode program. Kenapa di luar? Karena kamu ingin mengubah perilaku aplikasi tanpa harus mengedit kode sumbernya.

  • Konfigurasi fleksibel — bisa berubah tanpa mengubah kode
  • Disimpan terpisah — di file .env, Docker, Kubernetes, atau server
  • Dibaca saat runtime — aplikasi baca nilainya saat dijalankan: process.env.NAMA
  • Contoh umum — URL database, API key, port, mode aplikasi (dev/prod)
contoh-dasar.js
// Membaca env var dengan fallback (nilai default)
const port = process.env.PORT || 3000
const mode = process.env.NODE_ENV || 'development'

console.log(`Server jalan di port ${port} (mode: ${mode})`)

💡 Tips: Operator || artinya "kalau env var-nya kosong atau tidak ada, pakai nilai default setelahnya." Jadi aplikasi kamu tetap jalan meskipun env var belum diset!

⚙️ Environment Variables di Next.js

Di Next.js, ada aturan khusus soal env var yang wajib kamu pahami:

📁 File .env.local

Next.js otomatis membaca file .env.local di root project kamu. File ini untuk development dan TIDAK boleh di-commit ke Git!

🌐 Prefix NEXT_PUBLIC_

Variabel dengan prefix NEXT_PUBLIC_ bisa dibaca di browser (client-side). Tanpa prefix, variabel hanya bisa dibaca di server — ini untuk menjaga keamanan data sensitif.

  • NEXT_PUBLIC_APP_NAME → ✅ bisa di browser DAN server
  • DATABASE_URL → ✅ hanya di server, ❌ tidak di browser
  • API_SECRET_KEY → ✅ hanya di server, ❌ tidak di browser

💡 Inti konsepnya: Kode process.env.APP_MESSAGE itu sama di mana-mana. Yang berubah adalah nilainya — tergantung lingkungan tempat aplikasi dijalankan.

🔐 Kenapa Environment Variables Penting?

1. Security (Keamanan)

Jangan pernah tulis password, API key, atau secret langsung di kode. Kalau kode di-push ke GitHub dan repo-nya public, semua orang bisa lihat secret kamu!

2. Flexibility (Fleksibilitas)

Satu codebase, banyak environment. Mau deploy ke server lokal, Docker, atau cloud — kode tetap sama, yang berubah cuma env var.

3. Docker

Di Docker, kamu bisa pass env var pakai flag -e atau file .env. Ini bikin container kamu bisa dikonfigurasi tanpa rebuild image.

4. CI/CD

Di GitHub Actions, kamu bisa simpan secret di Settings → Secrets. Di Kubernetes, pakai ConfigMap atau Secret. Semua pakai konsep yang sama: env var.

⚠️ JANGAN PERNAH commit file .env atau .env.local ke Git! Selalu tambahkan ke file .gitignore supaya tidak ikut ter-upload.

💻 Contoh Kode

1. File .env.local

Buat file ini di root project Next.js kamu:

.env.local
# Variabel publik (bisa dibaca browser)
NEXT_PUBLIC_APP_NAME=Praktikum Cloud Native
NEXT_PUBLIC_APP_MESSAGE=Selamat datang!

# Variabel privat (HANYA server — jangan pakai NEXT_PUBLIC_)
DATABASE_URL=postgresql://user:pass@localhost/db
API_SECRET_KEY=rahasia-super-aman-123

2. Membaca di Server Component

Di Server Component, kamu bisa baca semua env var — baik yang public maupun private:

app/page.js — Server Component
// Server Component — bisa baca SEMUA env var
export default function Home() {
  // ✅ Variabel private — bisa dibaca di server
  const dbUrl = process.env.DATABASE_URL

  // ✅ Variabel public — juga bisa dibaca di server
  const appName = process.env.NEXT_PUBLIC_APP_NAME

  return (
    <div>
      <h1>{appName}</h1>
      <p>Database terhubung!</p>
    </div>
  )
}

3. Membaca di Client Component

Di Client Component (browser), kamu hanya bisa baca variabel dengan prefix NEXT_PUBLIC_:

components/Header.js — Client Component
'use client'

export default function Header() {
  // ✅ NEXT_PUBLIC_ bisa dibaca di browser
  const appName = process.env.NEXT_PUBLIC_APP_NAME

  // ❌ Ini akan undefined di browser!
  const secret = process.env.API_SECRET_KEY

  return (
    <header>
      <h1>{appName}</h1>
      {/* secret akan undefined — ini AMAN! */}
    </header>
  )
}

⚠️ Ingat: process.env.API_SECRET_KEY akan mengembalikan undefined di browser. Ini memang SENGAJA oleh Next.js — supaya secret kamu tidak bocor ke client!

4. Di API Route

API Route jalan di server, jadi bisa baca semua env var dengan aman:

app/api/info/route.js
import { NextResponse } from 'next/server'

export async function GET() {
  // ✅ API Route = server-side, aman baca secret
  const secret = process.env.API_SECRET_KEY

  // Gunakan secret untuk autentikasi, dll
  return NextResponse.json({
    message: process.env.NEXT_PUBLIC_APP_MESSAGE,
    status: 'ok'
  })
}

5. Default Values (Nilai Bawaan)

Selalu sediakan nilai default supaya aplikasi tetap jalan meskipun env var belum diset:

config.js
// Jika PORT tidak diset, pakai 3000
const port = process.env.PORT || 3000

// Jika NODE_ENV tidak diset, pakai 'development'
const mode = process.env.NODE_ENV || 'development'

// Jika APP_NAME tidak diset, pakai nama default
const appName = process.env.NEXT_PUBLIC_APP_NAME || 'My App'

6. Docker dengan Environment Variables

Saat menjalankan container Docker, kamu bisa pass env var tanpa rebuild image:

Terminal
# Cara 1: Langsung pakai flag -e
docker run -e APP_MESSAGE="Hello Docker!" -p 3000:3000 myapp

# Cara 2: Pakai file .env
docker run --env-file .env.production -p 3000:3000 myapp

# Cara 3: Beberapa env var sekaligus
docker run \
  -e DATABASE_URL="postgresql://prod-server/db" \
  -e API_SECRET_KEY="super-secret" \
  -e NODE_ENV="production" \
  -p 3000:3000 myapp

⚖️ Perbandingan — Public vs Private

Ini tabel perbandingan yang wajib kamu hafal saat bekerja dengan env var di Next.js:

Aspek NEXT_PUBLIC_* Tanpa prefix
Dibaca di browser ✅ Ya ❌ Tidak
Dibaca di server ✅ Ya ✅ Ya
Aman untuk secret ❌ Tidak aman ✅ Aman
Contoh nama NEXT_PUBLIC_APP_NAME DATABASE_URL
Contoh isi "Nama Aplikasi" "postgresql://..."
Kapan pakai Data non-sensitif yang perlu ditampilkan di UI API key, password, connection string

💡 Aturan mudahnya: Kalau datanya boleh dilihat siapa saja → pakai NEXT_PUBLIC_. Kalau rahasia → jangan pakai prefix, biarkan hanya server yang bisa baca.

⚠️ Kesalahan Umum

Empat kesalahan yang sering dilakukan mahasiswa saat bekerja dengan env var:

❌ Salah

1. Commit .env.local ke Git

Terminal — file .env ikut ter-commit
git add .
git commit -m "push semua file"
# File .env.local ikut ter-push ke GitHub! 😱
✅ Benar

Tambahkan .env.local ke .gitignore:

.gitignore
# Environment variables
.env
.env.local
.env.production
.env*.local
❌ Salah

2. Menggunakan NEXT_PUBLIC_ untuk secret

.env.local — JANGAN begini!
# ❌ Password bisa dilihat di browser!
NEXT_PUBLIC_DATABASE_PASSWORD=rahasia123
NEXT_PUBLIC_API_SECRET=super-secret-key
✅ Benar
.env.local — yang benar
# ✅ Tanpa prefix = hanya server yang bisa baca
DATABASE_PASSWORD=rahasia123
API_SECRET=super-secret-key
❌ Salah

3. Lupa restart dev server setelah ubah .env

Kamu sudah ubah isi .env.local, tapi kok nilainya tidak berubah? Itu karena Next.js hanya baca file .env saat pertama kali dijalankan!

✅ Benar

Setelah mengubah file .env.local, selalu restart dev server:

Terminal
# Hentikan dev server (Ctrl+C), lalu jalankan lagi
npm run dev
❌ Salah

4. Tidak set env vars di Docker/production

Kamu sudah deploy ke Docker tapi lupa set env var. Aplikasi crash karena DATABASE_URL bernilai undefined!

✅ Benar

Selalu pastikan env var sudah diset di setiap environment:

Terminal — set env var di Docker
docker run \
  -e DATABASE_URL="postgresql://prod/db" \
  -e API_SECRET_KEY="production-secret" \
  -p 3000:3000 myapp

✏️ Latihan

Coba kerjakan latihan berikut di project Next.js kamu:

1Buat file .env.local

Buat file .env.local di root project dengan 3 variabel: satu NEXT_PUBLIC_STUDENT_NAME (isi nama kamu), satu NEXT_PUBLIC_CAMPUS (isi nama kampus kamu), dan satu SECRET_NIM (isi NIM kamu — ini private!).

2Tampilkan di Halaman

Di app/page.js, tampilkan NEXT_PUBLIC_STUDENT_NAME dan NEXT_PUBLIC_CAMPUS di halaman. Pastikan data muncul di browser!

3Buat API Route dengan Variabel Private

Buat file app/api/student/route.js yang membaca SECRET_NIM dan mengembalikannya sebagai JSON response. Cek hasilnya di /api/student. Verifikasi bahwa NIM tidak bisa dibaca langsung di browser console.

💡 Bonus: Setelah selesai, coba jalankan docker run -e NEXT_PUBLIC_STUDENT_NAME="Docker Student" ... dan lihat apakah nama berubah. Ini membuktikan kekuatan env var!
